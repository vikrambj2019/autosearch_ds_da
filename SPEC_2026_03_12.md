**Date:** 2026-03-12
**Status:** Active Development
**Version:** 0.1.0

## 1. Vision

A natural language analytics platform where users ask human-like questions about their data and receive deep, insightful reports — not just raw numbers. The system understands data grain, writes code, runs it, critiques the results, goes deeper, and self-improves the quality of its analysis.

Built on the Claude Agent SDK (Python), the system uses:
- **MCP tools** for deterministic operations (data profiling, code execution, scoring)
- **AgentDefinitions** for specialized sub-agents (analyst, critic, reporter, evaluator)
- **A skill registry** for extensibility (add new analysis types without changing core code)
- **Company context layer** for domain-specific knowledge injection

## 3. Core Concepts

### 3.1 Grain Awareness

Grain is the observation unit of the data. For panel data (member x month), this determines:
- **Aggregation logic:** "average cost per member" = groupby(member).mean().mean(), NOT df.mean()
- **ML splits:** split by entity (member), not by row, to avoid leakage
- **Count semantics:** "how many members" = nunique(entity), not len(df)

The `profile_data` tool detects grain automatically:
- Identifies candidate ID columns (high cardinality, no nulls, non-numeric names)
- Identifies candidate time columns (date-like values, ordered)
- Tests uniqueness of (id, time) combinations
- Classifies as: cross-sectional, time-series, panel, or transaction-level

Grain context is injected into every agent prompt.

### 3.2 Analytics Self-Improvement Loop

```
ROUND 1: Answer the user's question
    → analyst agent writes + runs code, interprets results

ROUND 2: Critic evaluates + generates follow-up questions
    → "Does this make sense? What's driving this pattern?"
    → Produces 2 follow-up questions

ROUND 3: Answer follow-ups
    → analyst answers the critic's questions

ROUND 4: Compile report
    → reporter assembles original answer + follow-up insights

ROUND 5: Score the report (LLM-as-judge)
    → evaluator scores on: completeness, depth, context, actionability, data_quality
    → If composite < threshold (0.70): loop back with improvement suggestion
    → If composite >= threshold: deliver final report
```

### 3.3 Skill Registry

Each analysis capability is a self-contained skill package:

```
registry/<skill_name>/
├── manifest.json    ← routing rules, column requirements, scoring weights
├── prompt.md        ← agent system prompt (supports {GRAIN_CONTEXT} placeholder)
├── patterns.md      ← (optional) code patterns the agent should know
├── template.py      ← (optional) starter code for complex analyses
├── examples.md      ← (optional) example questions + expected output shapes
└── scoring.json     ← (optional) custom scoring weights for evaluator
```

New skills are discovered at startup — no core code changes needed.

### 3.4 Company Context Layer

```
company_context/
├── coding_standards.md        ← code style rules injected into code-writing agents
├── domain_knowledge.md        ← domain-specific definitions and business rules
├── metrics_definitions.md     ← what each metric means in this organization
├── ml_best_practices.py       ← company's gold-standard ML template
└── report_style.md            ← report formatting preferences
```

Injected into every agent prompt automatically.

## 4. MCP Tools Specification

### 4.1 profile_data

**Purpose:** Profile a dataset — schema, grain detection, statistics, sample rows.

**Input:**
```json
{
  "data_path": "string — path to CSV file",
  "sample_rows": "int — number of sample rows to return (default: 5)"
}
```

**Output:** JSON with:
- `shape`: [rows, cols]
- `columns`: list of {name, dtype, n_null, pct_null, n_unique, top_values, min, max, mean, std}
- `grain`: {entity_col, time_col, grain_type, n_entities, n_periods, is_balanced, aggregation_guidance}
- `sample`: markdown table of sample rows
- `memory_mb`: float
- `warnings`: list of data quality issues

**Grain detection algorithm:**
1. Score each column as potential entity key: high cardinality + no nulls + non-numeric-looking
2. Score each column as potential time dimension: parseable as date + ordered + limited unique values
3. Test (entity, time) uniqueness → panel if unique, transaction if not
4. If no time column → cross-sectional
5. If no entity column → time-series aggregate

### 4.2 run_code

**Purpose:** Execute pandas analysis code in a sandboxed namespace.

**Input:**
```json
{
  "code": "string — pandas code to execute",
  "data_path": "string — path to CSV file",
  "return_format": "string — 'table' | 'scalar' | 'json' | 'markdown'"
}
```

**Execution environment:**
- Pre-loaded: `df` (the CSV), `pd`, `np`, `stats` (scipy.stats)
- NOT available: os, subprocess, sys, shutil, importlib, __import__
- Timeout: 30 seconds
- Memory: results truncated at 500 rows

**Output:** JSON with:
- `success`: bool
- `result`: formatted result (string)
- `result_shape`: [rows, cols] if DataFrame
- `execution_time_ms`: int
- `error`: traceback string if failed

### 4.3 validate_cols

**Purpose:** Fuzzy-match user's column references to actual column names.

**Input:**
```json
{
  "data_path": "string — path to CSV file",
  "user_columns": "string — comma-separated column names from user query"
}
```

**Output:** JSON with:
- `matches`: {user_ref: actual_col_name} mapping
- `unmatched`: list of user references that couldn't be matched
- `suggestions`: for unmatched, closest candidates

**Matching strategy:**
1. Exact match (case-insensitive)
2. Substring match (user says "county" → "COUNTY_LATEST")
3. Token overlap ("ER COST" → "MONTHLY_ER_COST")
4. Common abbreviation expansion ("avg" → "AVG", "er" → "ER")

## 5. Agent Specifications

### 5.1 Descriptive Analyst
- **Triggers:** distribution, average, count, total, percentile, how many, median, ratio
- **Tools:** profile_data, run_code, validate_cols
- **Model:** sonnet
- **Key rule:** Grain-aware aggregation. NEVER compute member stats on panel rows directly.

### 5.2 Diagnostic Analyst
- **Triggers:** why, correlation, significant, change, transition, anomaly, exceeds, despite, volatility
- **Tools:** profile_data, run_code, validate_cols
- **Model:** sonnet
- **Key rule:** Report test statistics + p-values. State significance level. Correlation != causation.

### 5.3 Critic
- **Purpose:** Evaluates an answer, sanity-checks it, generates 2 follow-up questions
- **Tools:** profile_data (for schema reference, no code execution)
- **Model:** sonnet
- **Output format:** JSON with sanity_check, follow_up_questions, reasoning

### 5.4 Reporter
- **Purpose:** Compiles analysis rounds into a structured insight report
- **Tools:** none (pure text synthesis)
- **Model:** sonnet
- **Output:** Executive summary format with key finding, supporting details, caveats

### 5.5 Evaluator
- **Purpose:** Scores a report on 5 dimensions, returns composite score
- **Tools:** none
- **Model:** sonnet
- **Scoring:** 0.25*completeness + 0.25*depth + 0.20*context + 0.20*actionability + 0.10*data_quality
- **Threshold:** 0.70 (configurable per skill)
- **Output:** JSON with per-dimension scores + composite + improvement_suggestion
